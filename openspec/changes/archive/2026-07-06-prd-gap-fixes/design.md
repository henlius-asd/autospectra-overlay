## Context

当前代码库已实现核心三栏布局、文件解析、Waterfall 图表渲染、自动对齐、持久化和撤销/重做。但基于 PRD 差距分析，仍有 6 项功能缺失和 1 个关键 Bug。这些修改集中在 UI 交互层（左栏曲线管理、右栏元数据展示、大括号工具）和 Store 数据模型扩展。

现有架构遵循 Zustand 状态驱动 + 组件化模式，所有修改均在现有架构框架内进行，不引入新依赖。

## Goals / Non-Goals

**Goals:**
- 曲线列表支持右键上下文菜单操作
- 左栏区分叠图区与未叠图区，支持拖拽排序
- 基准线以 ★ 标识可视化，默认曲线 #1 为基准线
- 修复大括号创建 Bug，提供工具按钮触发创建
- 曲线支持原始名称（唯一标识）与显示名称（可编辑别名）
- 左栏支持按名称搜索/筛选曲线
- 右侧栏上方展示选中曲线的文件元数据

**Non-Goals:**
- 不实现 Tooltip 真实值还原（实验员自行分析）
- 不实现单线 X/Y 微调控件（之前实现过但不符需求）
- 不实现单曲线 CSV 导出（暂缓）
- 不修复一般性缺失（IndexedDB 降级、颜色一致性等 8 项）

## Decisions

### 1. 叠图区/未叠图区数据模型

**决策**：在 `curveStore` 中新增 `stagingOrder: string[]` 数组，存储已进入叠图区的曲线 ID 列表（有序）。`addCurves` 导入的曲线默认不在 `stagingOrder` 中（处于未叠图区）。用户勾选可见性（checkbox）时，曲线自动进入 `stagingOrder` 末尾。取消勾选时从 `stagingOrder` 移除。

**替代方案**：使用独立的 `staging` flag 字段 —— 放弃，因为还需要维护顺序，数组更自然。

**理由**：`stagingOrder` 数组既表示"哪些曲线在叠图区"，也表示"它们的渲染顺序"。未在数组中的曲线位于未叠图区。这比单独维护一个 boolean + 一个 order 数组更简洁。

### 2. 拖拽排序实现

**决策**：使用 HTML5 原生 Drag and Drop API（`draggable` + `onDragStart/onDragOver/onDrop`），不引入第三方拖拽库。

**理由**：拖拽需求简单（仅在同列表内排序），不需要跨容器拖拽或复杂动画。原生 API 足够，避免增加 bundle 体积。

### 3. 基准线渲染位置

**决策**：基准线（曲线 #1）在 Waterfall 图中始终使用 `layerYOffset = 0`（即位于图表最下侧），其他曲线按 `stagingOrder` 顺序依次叠加 Y 偏移。

**理由**：用户明确要求"基准线默认位于曲线 #1 位于图表的最下侧"，而不是让基准线视觉上位于中间。

### 4. 大括号创建方式

**决策**：移除 Shift+拖拽交互（因 SVG `pointer-events` Bug 导致死锁），改为在工具栏或右侧面板提供"插入大括号"按钮。点击后进入"大括号放置模式"，在图表上点击两个端点创建大括号，弹出输入框填写标签。

**理由**：修复 Shift+拖拽的 `pointer-events: none` 死锁问题需要重新设计事件处理。使用工具按钮模式更直观、更可靠，且与用户"加入一个工具专门来插入大括号"的建议一致。

### 5. 曲线别名数据模型

**决策**：`CurveData` 接口新增 `displayName?: string` 可选字段。当 `displayName` 存在时，图表和曲线列表使用 `displayName`；否则回退到 `name`（原始名称/SampleName）。原始名称 `name` 作为唯一标识符不可修改。

**理由**：最小侵入性——不破坏现有 `name` 字段的语义，Parser 不需要修改，仅在 UI 层增加覆盖逻辑。

### 6. 元数据展示位置

**决策**：在 `uiStore` 中新增 `selectedCurveId: string | null`。右侧栏顶部新增 `MetadataPanel` 组件，显示 `selectedCurveId` 对应曲线的 `metadata` 字段。用户通过点击曲线列表中的曲线行来选中曲线（不同于勾选可见性）。

**理由**：右侧栏上方是自然的信息展示位置，切换曲线通过点击曲线列表实现，不需要额外的下拉选择器。

### 7. 搜索/筛选实现

**决策**：在 `LeftPanel` 中维护本地 `filterText` 状态，对曲线列表进行客户端过滤。不存入 Store（临时 UI 状态）。

**理由**：搜索是纯 UI 行为，不影响数据模型，不需要持久化或跨组件共享。

## Risks / Trade-offs

- **[风险] 拖拽排序的移动端兼容性**：HTML5 DnD API 在触屏设备上支持有限 → 本工具仅支持桌面端（PRD 明确），无影响
- **[风险] stagingOrder 与 visibleCurves 状态不一致**：如果直接操作 `visibleCurves` 而不更新 `stagingOrder`，两个状态可能不同步 → 统一通过 `toggleCurveVisibility` action 管理，内部同时更新两个状态
- **[风险] 大括号工具按钮的交互流程**：用户需要学习新的按钮操作方式 → 提供清晰的按钮 tooltip 和操作提示
